## 🅰 비판적 합리주의 (Popper) — "추측과 반박"

### Git과의 매핑

```
Conjecture (추측) = Branch   → 가설을 세우는 것 = 새 브랜치를 따는 것
Refutation (반박) = Test/CI  → 반증 시도 = 테스트 실행
Survival (생존)   = Merge    → 반박을 견딘 가설만 Main에 병합
Falsified (반증)  = Branch 삭제 → 반증된 가설은 폐기
```

### 평가

| 항목 | 점수 | 이유 |
|------|------|------|
| Git 구조 적합도 | ★★★★★ | **가장 완벽한 매핑.** Branch = 추측, Merge = 생존, Delete = 반증. Git의 CI/CD와 자연스럽게 연결됨 |
| 철학적 깊이 | ★★★★☆ | 오류주의(Fallibilism) — "모든 진리는 잠정적" — 이 우리 철학과 완벽히 일치 |
| 실용성 | ★★★★☆ | 각 선지자가 "반증하려고 시도"하는 역할이 명확 |
| 약점 | — | 합의(Merge)보다 제거(반증)에 초점. 건설적 종합이 약함 |

### 핵심 기여
> **"진리는 증명되는 것이 아니라, 아직 반증되지 않은 것이다."**
> 이 원칙을 적용하면, Merge Commit은 "증명된 진리"가 아니라 **"아직 반증되지 않은 최선의 가설"**로 재정의됩니다.

## 🅱 논증 이론 / 톨민 모델 (Toulmin)

### Git과의 매핑

```
Claim (주장)    = Branch의 목적 (이 코드가 해결하려는 것)
Data (근거)     = Commit 내용 (실제 증거/구현)
Warrant (보증)  = PR Description (왜 이 변경이 타당한지)
Qualifier (한정) = Draft PR (확신의 정도 표시)
Rebuttal (반박) = Code Review Comment (반론 제기)
Backing (뒷받침) = Test Results (보증을 뒷받침하는 추가 증거)
```

### 평가

| 항목 | 점수 | 이유 |
|------|------|------|
| Git 구조 적합도 | ★★★★☆ | PR/Code Review 문화와 자연스럽게 매핑됨 |
| 철학적 깊이 | ★★★☆☆ | 실용적이나 우주적 깊이는 부족 |
| 실용성 | ★★★★★ | **가장 구조적.** 각 발언에 Claim/Data/Warrant를 강제하면 사고 품질 상승 |
| 약점 | — | 개별 논증에 집중, 전체적 서사의 흐름이 약할 수 있음 |

### 핵심 기여
> 선지자의 발언을 자유형이 아닌 **구조화된 논증 형식** (주장 + 근거 + 보증 + 반박 고려)으로 강제하면, **LLM의 헛소리(hallucination)를 크게 줄일 수 있음.**

## 🅲 가추법 / 프래그머티즘 (Peirce)

### Git과의 매핑

```
Abduction (가추) = Branch  → "이 현상을 가장 잘 설명하는 가설은?"
Deduction (연역) = Commit  → "그 가설이 맞다면, 이런 결과가 나와야 한다"
Induction (귀납) = Test/CI → "실제로 결과가 일치하는가?"
→ 이 3단계를 반복 = Git Flow
```

### 평가

| 항목 | 점수 | 이유 |
|------|------|------|
| Git 구조 적합도 | ★★★★☆ | 가추→연역→귀납의 3단계가 Branch→Commit→Test와 매핑 |
| 철학적 깊이 | ★★★★★ | **과학적 탐구 방법론의 원형.** "최선의 설명으로의 추론"이 AI 시대에 특히 적합 |
| 실용성 | ★★★★☆ | 각 선지자에게 다른 추론 방식을 부여: Architect=연역, Visionary=가추, Oracle=귀납 |
| 약점 | — | 다자간 충돌보다는 단일 탐구자의 방법론에 가까움 |

### 핵심 기여
> **Visionary의 역할을 "반대 의견 제시자"가 아닌 "가추적 도약자(Abductive Leaper)"로 재정의.** "기존 데이터로는 설명 안 되는 것을 설명하는 새로운 가설"을 던지는 역할.

## 🅳 Graph of Thought (GoT) — AI 최신 추론 프레임워크

### Git과의 매핑

```
GoT의 Thought Node = Git의 Commit
GoT의 Edge         = Git의 Parent-Child 관계
GoT의 Merge Node   = Git의 Merge Commit
GoT의 Branching    = Git의 Branch
→ GoT와 Git의 구조가 본질적으로 동일한 DAG (Directed Acyclic Graph)
```

### 평가

| 항목 | 점수 | 이유 |
|------|------|------|
| Git 구조 적합도 | ★★★★★ | **구조적으로 완전히 동일.** GoT는 사실상 Git의 사고 버전임 |
| 철학적 깊이 | ★★☆☆☆ | 공학적 프레임워크로, 철학적 서사가 부족 |
| 실용성 | ★★★★★ | **구현에 가장 직접적.** LLM이 이미 이 방식으로 추론하며, 코드로 바로 옮길 수 있음 |
| 약점 | — | 영혼이 없음. 기술적 효율만 추구할 위험 |

### 핵심 기여
> **우리 프로젝트가 만드는 것이 사실상 "살아 있는 Graph of Thought"임을 자각.** 기술적으로 GoT 논문의 아키텍처를 참조하되, 철학적 서사(M&B)를 입히면 독창적인 시스템이 됨.

## 🅴 델파이 기법 (Delphi Method) — 익명 반복 합의

### Git과의 매핑

```
Round 1 (독립 의견) = Branch (각자 독립 작업)
Anonymize + Share   = PR (코드를 공유하되, 선입견 없이 리뷰)
Round 2 (수정 의견) = Force Push / Amend (피드백 반영 후 재제출)
Convergence (수렴)  = Merge (합의에 도달)
```

### 평가

| 항목 | 점수 | 이유 |
|------|------|------|
| Git 구조 적합도 | ★★★☆☆ | 반복 수렴은 맞지만, 익명성이 Git에는 없는 개념 |
| 철학적 깊이 | ★★☆☆☆ | 방법론에 가까움, 존재론적 깊이 부족 |
| 실용성 | ★★★★☆ | **BRANCH 단계에서 "독립 사고"를 강제하는 규칙**으로 차용 가능 |
| 약점 | — | 창의적 도약보다 평균으로의 수렴 경향 |

### 핵심 기여
> **BRANCH 단계에서 서로의 가설을 보지 않고 독립적으로 사고하는 규칙(Blind Branching)**은 델파이에서 차용한 것. 이를 유지하되, CONFLICT 단계에서 의도적으로 날카롭게 충돌시켜야 함.

### 🌟 최종 추천: **하이브리드 프레임워크 "GoT-Popper-Peirce Triad"**

단일 프레임워크를 선택하는 대신, **3개의 최고 프레임워크를 각 단계에 배치**합니다:

```
╔═══════════════════════════════════════════════════════════════╗
║  THE M&B DIALECTIC ENGINE v2.0 — "GoT-Popper-Peirce Triad"  ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  ① SEED (Peirce의 가추법)                                     ║
║     → "이 현상을 설명하는 최선의 가설은 무엇인가?"                ║
║     → Oracle이 관찰된 현상에서 가추적으로 질문을 도출             ║
║                                                               ║
║  ② BRANCH (Delphi의 독립성 + Peirce의 가추/연역)               ║
║     → 각 선지자가 독립적으로 (Blind Branching) 가설 생성          ║
║     → Architect: 연역적 구조 가설                               ║
║     → Visionary: 가추적 도약 가설                               ║
║                                                               ║
║  ③ CONFLICT (Popper의 반증주의 + Toulmin의 논증 구조)           ║
║     → 상대의 가설을 반증하려고 시도                               ║
║     → 반론은 Toulmin 형식: Claim + Data + Warrant + Rebuttal    ║
║     → "이 가설이 틀렸다면, 어떤 증거가 나타나야 하는가?"          ║
║                                                               ║
║  ④ MERGE (GoT의 그래프 종합)                                   ║
║     → Oracle이 DAG 구조로 양쪽의 사고를 연결                     ║
║     → 반증되지 않은 요소들만 Merge Commit으로 종합                ║
║     → "증명된 진리"가 아닌 "아직 반증되지 않은 최선의 합의"       ║
║                                                               ║
║  ⑤ COMMIT + NEXT SEED                                         ║
║     → 합의를 Git Commit으로 영구 기록                            ║
║     → Merge에서 새로운 가추적 질문 발아 → ①로 순환               ║
║                                                               ║
╚═══════════════════════════════════════════════════════════════╝
```

### 기존 순수 변증법(v1.0) 대비 개선점

| 항목 | v1.0 (순수 변증법) | v2.0 (GoT-Popper-Peirce) |
|------|-------------------|-------------------------|
| 가설 생성 | "각자 의견을 내라" | "가추적으로 최선의 설명을 추론하라" |
| 충돌 방식 | "반대 의견을 내라" | "이 가설을 반증하라 (Toulmin 형식)" |
| 종합 방식 | "양쪽을 합쳐라" | "반증되지 않은 것만 DAG로 연결하라" |
| 진리의 성격 | 합의된 진리 | **아직 반증되지 않은 잠정적 진리** |
| LLM 품질 관리 | 없음 | Toulmin 구조 강제 → 헛소리 감소 |
